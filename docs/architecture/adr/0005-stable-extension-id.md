# ADR-0005: Extension ID יציב באמצעות שדה `key` מוזרק

- סטטוס: Accepted
- תאריך: 2026-06-08

## הקשר
ל-unpacked extension, Chrome גוזר את ה-Extension ID מ-`key` ב-manifest אם קיים, אחרת מנתיב התיקייה (לא יציב בין מחשבים). דרוש ID זהה בין גרסאות ובין מחשבים — לצורך Native Messaging `allowed_origins`, externally_connectable, ועקביות נתונים.

## החלטה
- לכל פרויקט נוצר זוג מפתחות **RSA-2048** ("project key") פעם אחת, בשירות החתימה. **המפתח הפרטי נשמר בשרת בלבד** (טבלת `project_keys`, מוצפן at-rest), לעולם לא נשלח ללקוח ולא נכנס ל-ZIP.
- שדה ה-`key` ב-manifest הוא ה-**מפתח הציבורי** של אותו זוג בבסיס64 (DER SubjectPublicKeyInfo). Chrome מחשב ממנו ID יציב ודטרמיניסטי.
- ה-Worker **מזריק** את `key` ל-`manifest.json` בעת אריזת ה-Artifact המאושר (לא משנה את ה-ZIP המקורי שהמפתח העלה). כך אותו פרויקט מקבל את אותו `key` ⇒ אותו ID בכל גרסה ובכל מחשב.
- ה-Extension ID מחושב גם בשרת (אותו אלגוריתם של Chrome: SHA-256 של ה-DER public key, 32 התווים הראשונים ממופים a–p) ונשמר על הפרויקט, כדי ש-`allowed_origins` ו-UI יציגו אותו עוד לפני התקנה.
- **שינוי Extension ID בין גרסאות = חסום** (סעיף 15).

## אובדן/שחזור מפתח
- גיבוי `project_keys` מוצפן הוא קריטי. אובדן המפתח הפרטי = אובדן היכולת לשמור ID יציב (התוסף ייראה ל-Chrome כתוסף חדש). תהליך שחזור ותרחיש אובדן מתועדים ב-[docs/security/extension-id.md].

## בדיקות
- אוטומטי: בהינתן אותו public key, חישוב ה-ID זהה (Python ↔ .NET ↔ ערך הצפוי של Chrome) — `tests/`.
- ידני (חובה ב-Chrome): טעינת שתי גרסאות עם אותו `key` נותנת אותו ID; טעינה בשני מחשבים נותנת אותו ID. ראו manual-chrome-tests פריטים 1–2.

## תוצאה
ID יציב, מאומת, בלי לחשוף סוד ללקוח. תלוי בגיבוי מאובטח של `project_keys`.

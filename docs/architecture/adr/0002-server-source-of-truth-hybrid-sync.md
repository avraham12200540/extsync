# ADR-0002: השרת כמקור אמת + סנכרון היברידי (WS + Polling)

- סטטוס: Accepted
- תאריך: 2026-06-08

## הקשר
ה-Agent רץ במכונת המשתמש, לעיתים לא מחובר. השרת מנהל פרויקטים, גרסאות, ערוצים, rollout וחתימות. צריך מודל סנכרון אמין שעובד גם אופליין וגם בזמן אמת, בלי כפילות עדכונים.

## החלטה
חלוקת אחריות מפורשת:

- **השרת = מקור אמת** ל: פרויקטים, גרסאות, ערוצים, גרסה פעילה לכל ערוץ, rollout %, קישורי התקנה, חתימות, סטטוס גרסה, משתמשים/צוותים.
- **ה-Agent = מקור אמת מקומי** ל: אילו תוספים מותקנים במחשב, התיקייה המקומית, הגרסה בפועל, סטטוס העדכון האחרון, האם Chrome טען, האם נדרש reload, היסטוריית rollback מקומית, לוגים מקומיים.

מודל היברידי:
1. כש-ה-Agent פתוח — WebSocket מאובטח (`WS /agent/events`), עם reconnect ו-exponential backoff.
2. פרסום גרסה → השרת דוחף `update_available` ל-Agents הרלוונטיים (לפי rollout).
3. גם בלי WS — בדיקה מחזורית (`POST /agent/check-updates`).
4. Scheduled Task ב-Windows מריץ בדיקה בכניסה ובמרווחים (ברירת מחדל: כל 4 שעות).
5. **כל פעולה Idempotent** — מפתח `(projectId, releaseId, sequence)`; אותו עדכון לא מותקן פעמיים.
6. אחרי ניתוק — סנכרון משלים כשהחיבור חוזר.

מצב גרסה נישא ב-`sequence` מונוטוני עולה ב-metadata. ה-Agent מתעלם מ-sequence ישן, אלא אם זה Rollback חתום.

## חלופות שנשקלו
- **Polling בלבד**: פשוט אך עדכונים איטיים (עד 4 שעות). נדחה כברירת מחדל אך נשמר כ-fallback.
- **Push בלבד (WS)**: לא עמיד לניתוקים/Agent סגור. נדחה כפתרון יחיד.

## תוצאה
זמן-אמת כשאפשר, עמידות כשלא. Idempotency ו-sequence מונעים מצבי מרוץ וכפילויות. ראו ADR-0003 (חתימה) ו-ADR-0004 (החלפת קבצים).

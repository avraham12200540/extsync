# ADR-0003: חתימת Metadata ב-Ed25519 + אימות Artifact ב-SHA-256

- סטטוס: Accepted
- תאריך: 2026-06-08

## הקשר
ה-Agent חייב לסרב להתקין קוד שלא הגיע מהפלטפורמה ולא שונה בדרך. דרושה שרשרת אמון: השרת חותם, ה-Agent מאמת עם מפתח ציבורי מוטמע.

## החלטה
- **חתימה: Ed25519** על ה-Release Metadata הקנוני (ראו `packages/release-schema`). מהיר, חתימות קצרות, מימוש בטוח בכל השפות (PyNaCl/cryptography ב-Python, NSec/BouncyCastle ב-.NET, `@noble/ed25519` ב-TS).
- **שלמות Artifact: SHA-256** של ה-ZIP, נכלל ב-metadata וחתום יחד איתו. ה-Agent מוריד, מוודא גודל, מחשב SHA-256, ורק אז בודק חתימה.
- **קנוניזציה**: JSON קנוני דטרמיניסטי (מפתחות ממוינים, ללא רווחים, UTF-8, ללא הצגת שדה `signature` עצמו) — מוגדר פעם אחת ב-`release-schema` ומיושם זהה ב-Python/TS/.NET. ראו [docs/security/signing.md].
- **הפרדת שירותים**: שירות החתימה (`extsync_signing`) מבודד רשתית, מחזיק את המפתח הפרטי מחוץ ל-DB, מקבל **רק metadata מוגדר** (לא קבצי משתמש), מתעד כל חתימה, ודורש token פנימי.
- **Key rotation**: `keyId` ב-metadata; ה-Agent מחזיק מספר מפתחות ציבוריים במקביל; מפתח שנפרץ נחסם.

ה-Agent מסרב להתקין כאשר: חתימה לא תקינה / hash לא תואם / projectId שונה / הגרסה לא מאושרת לערוץ / sequence נמוך (אלא אם Rollback חתום) / קובץ גדול מהמוצהר / נדרש Agent חדש יותר / כשל בדיקה מקומית.

## חלופות
- **RSA-2048 חתימות**: גדולות ואיטיות יותר; Ed25519 מודרני ומספיק.
- **HMAC**: סוד משותף — לא מתאים לאימות צד-לקוח (היה חושף את הסוד ב-Agent). נדחה. (HMAC כן משמש ל-Webhooks, שם הסוד נשאר בצד השרת והלקוח של המפתח.)

## תוצאה
ה-Agent מאמת מקור ושלמות לפני כל התקנה. סודות חתימה לעולם לא בקוד הצד-לקוח ולא ב-ZIP.

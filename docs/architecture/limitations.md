# מגבלות ידועות (Chrome / Windows) — קראו לפני הכול

מסמך זה מפרט את המגבלות **האמיתיות** שאי אפשר לעקוף בצורה נתמכת. ExtSync תוכננה *סביב* המגבלות האלה, לא נגדן. כל פתרון בקוד מתייחס למגבלה מכאן.

## 1. אין התקנה שקטה של תוסף לא-ארוז

ב-Chrome רגיל (לא Chrome for Business עם מדיניות ארגונית, לא ChromeOS מנוהל), **אין API נתמך** שמאפשר לתהליך חיצוני להתקין תוסף בלי:

- חנות Chrome Web Store, **או**
- מדיניות ארגונית (`ExtensionInstallForcelist` דרך Group Policy / Registry / MDM), **או**
- פעולה ידנית של המשתמש ב-`chrome://extensions`.

ExtSync לא משתמשת בחנות ולא מתחזה למדיניות ארגונית. לכן **ההתקנה הראשונה דורשת ארבעה צעדים ידניים מהמשתמש**:

1. פתיחת `chrome://extensions`
2. הפעלת "מצב מפתח" (Developer mode)
3. לחיצה על "טען פריט לא ארוז" (Load unpacked)
4. בחירת התיקייה ש-ExtSync Agent הכין

מה ש-ExtSync Agent **כן** עושה כדי לצמצם חיכוך: פותח את דף התוספים, מעתיק את נתיב התיקייה ללוח, פותח את התיקייה ב-Explorer, מציג הדרכה חזותית, ואז מזהה שהתוסף נטען (דרך ה-Bridge / Native Messaging).

מה שאסור ולא נעשה: שינוי קבצים פנימיים של Chrome, זיוף מדיניות ארגונית, "הזרקת" תוסף, או כל ניסיון להסיר את הצורך במצב מפתח.

## 2. עדכון = החלפת קבצים + טעינה מחדש

אחרי שהתוסף נטען כ-unpacked מתיקייה, Chrome קורא את הקבצים מאותה תיקייה. ExtSync Agent מעדכן ע"י **החלפה בטוחה של תוכן התיקייה** ואז גורם ל-reload:

- אם התוסף כולל **ExtSync Bridge** → ה-Bridge מקבל הודעת `reload_required` מאומתת דרך Native Messaging ומריץ `chrome.runtime.reload()`. העדכון נכנס לתוקף מיד.
- אם **אין Bridge** או החיבור נכשל → ExtSync מסמן את המצב כ-`reload_required` (Pending Restart). העדכון יושלם כשהמשתמש יטען מחדש ידנית או יפעיל מחדש את Chrome. **אין דרך נתמכת** לכפות reload על תוסף ללא Bridge.

מגבלת קבצים נעולים: בזמן ש-Chrome פתוח הוא עשוי להחזיק handle לקבצים בתיקייה. ראו ADR-0004 על אסטרטגיית ההחלפה (rename של תיקיות עם fallback ל-journaled copy). אם קובץ נעול — **לא מוחקים את הגרסה הפעילה**, מנסים שוב, ומסמנים Pending Restart.

## 3. `chrome.runtime.reload()` בלבד — אין reload של תוסף אחר

ה-Bridge יכול להפעיל reload רק על **התוסף שמכיל אותו**. אין API לתוסף אחד לטעון מחדש תוסף אחר. לכן כל תוסף מנוהל חייב לכלול את ה-Bridge בעצמו כדי לקבל reload אוטומטי.

## 4. Extension ID יציב = שדה `key` ב-manifest

ב-unpacked, Chrome מחשב את ה-Extension ID מתוך שדה `key` ב-manifest (מפתח ציבורי RSA בבסיס64). בלי `key`, ה-ID נגזר מנתיב התיקייה ולכן **משתנה בין מחשבים/נתיבים**. ExtSync מזריקה `key` קבוע (נגזר משירות החתימה/הפרויקט) כדי לשמור ID יציב בין גרסאות ומחשבים. המפתח הפרטי שמייצר את ה-`key` **לעולם** לא נשלח ללקוח או נכנס ל-ZIP. ראו ADR-0005 ו-[docs/security/extension-id.md](../security/extension-id.md).

## 5. אין הורדת קוד מרוחק

מדיניות Manifest V3 ומדיניות ExtSync: כל קוד התוסף חייב להיכלל בחבילת הגרסה החתומה. ה-Bridge **לא** מוריד קוד, **לא** מריץ קוד מהשרת, ו-`chrome.runtime.reload()` מתבצע רק אחרי הודעה חתומה/מאומתת מה-Agent המקומי.

## 6. Native Messaging דורש רישום מדויק של origins

`allowed_origins` ב-manifest של ה-Native Host חייב להכיל את ה-Extension ID המדויק (`chrome-extension://<id>/`). מכיוון שה-ID ידוע רק אחרי שהתוסף נרשם, **ה-Agent מעדכן את ה-host manifest** עם ה-origin בעת רישום תוסף חדש. ראו [docs/architecture/adr/0006-native-messaging-hkcu.md].

## 7. הרשאות מוגברות דורשות אישור משתמש

Chrome עצמו ידרוש מהמשתמש לאשר מחדש תוסף כשהרשאות "מסוכנות" משתנות בטעינה מחדש מסוימת. ExtSync לא מנסה לעקוף זאת: כשגרסה חדשה מוסיפה הרשאה רגישה / גישה לכל האתרים / nativeMessaging, ה-Agent **עוצר** ומציג למשתמש Diff ברור לפני התקנה (סעיף 15 באפיון).

## 8. התקנה ברמת המשתמש

ה-Agent, ה-Native Host וה-Registry של `extsync://` ושל ה-host מותקנים תחת `HKCU` ו-`%LOCALAPPDATA%` — ללא צורך בהרשאות מנהל ברוב המקרים. זה גם אומר שההתקנה היא per-user (לא per-machine).

---

תקציר: ExtSync לא "פורצת" את Chrome. היא מנהלת את מחזור החיים סביב מגבלת ה-unpacked: התקנה ראשונה ידנית מודרכת, ואז עדכונים אוטומטיים בטוחים, חתומים, עם Rollback.
